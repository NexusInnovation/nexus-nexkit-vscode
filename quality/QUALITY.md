## Purpose

Quality for this extension means repeatable workspace automation without silent data corruption, hidden lifecycle side effects, or command drift between declaration and runtime behavior. This is a tooling product: failures here amplify downstream development failures.

Deming applies directly: quality is built into startup/activation contracts, managed file safety, and shell-execution boundaries before shipping. Juran's fitness-for-use means a user can open VS Code, run initialization, and trust that configuration updates are safe and deterministic. Crosby's quality-is-free principle applies because preventing config wipeouts and command contract regressions is significantly cheaper than incident triage after release.

## Coverage Targets

| Subsystem                         | Target | Why                                                                                                                      |
| --------------------------------- | ------ | ------------------------------------------------------------------------------------------------------------------------ |
| Activation lifecycle              | 90%+   | Duplicate process handlers and stale command registration failures are high-impact and difficult to detect post-release. |
| Managed file watcher              | 90%+   | Rollback logic mutates user files and can corrupt content if byte fidelity is not preserved.                             |
| MCP configuration merge           | 90%+   | Parse-failure fallback can destroy user config state; this is a data-loss class risk.                                    |
| Workflow runner command builder   | 85%+   | Shell argument boundary failures can run wrong jobs or fail unpredictably across OS.                                     |
| Conversion subprocess integration | 80%+   | Sync I/O and subprocess controls affect responsiveness and stability of extension host.                                  |
| Test contract enforcement         | 85%+   | Skipped stale tests create CI blind spots where regressions ship silently.                                               |

## Coverage Theater Prevention

- A test that checks command id presence in one list but never validates runtime registration is theater.
- A test that asserts workflow command string exists but never validates argument-boundary safety with edge inputs is theater.
- A test that confirms watcher prompts appear but never validates byte-exact rollback behavior is theater.
- A conversion test that only checks no throw without asserting timeout/cleanup behavior is theater.

## Fitness-to-Purpose Scenarios

### Scenario 1: Reloaded Host Duplicate Fault Streams

Requirement tag: [Req: inferred — from src/extension.ts handler lifecycle]
What happened: If global process handlers are added on each activate path and never cleaned, one runtime fault can emit duplicate telemetry/log records, obscuring incident timelines.
The requirement: Handler registration is idempotent across host reload and reactivation paths.
How to verify: Simulate activation twice and assert one handler per event type.

### Scenario 2: Managed File Non-UTF8 Rollback Corruption

Requirement tag: [Req: inferred — from NexkitFileWatcher rollback behavior]
What happened: Watcher rollback currently assumes UTF-8 text. A non-UTF8 managed asset can be rewritten with altered bytes after external edit/delete handling.
The requirement: Rollback and restore paths preserve byte-exact original content.
How to verify: Seed managed file with non-UTF8 bytes, trigger change/delete path, and assert byte equality after restore.

### Scenario 3: MCP Parse-Failure Data Loss

Requirement tag: [Req: inferred — from MCP config merge fallback]
What happened: A malformed edit in mcp.json can trigger parse failure and reset to empty config before write, erasing unrelated server definitions.
The requirement: Parse failure must abort destructive writes and preserve existing file bytes.
How to verify: Provide malformed JSON with existing entries, invoke add operation, and assert file unchanged with explicit error.

### Scenario 4: Shell Boundary Drift in Workflow Execution

Requirement tag: [Req: inferred — from OS-specific command builders]
What happened: Job/event values containing quotes or control chars can alter shell argument boundaries when interpolated into command strings.
The requirement: Command parameter encoding must be shell-safe and equivalent across PowerShell and POSIX builders.
How to verify: Pass adversarial event/job strings and assert target script receives exact intended arguments.

### Scenario 5: Startup Prompt Intrusion

Requirement tag: [Req: inferred — from startup verification auth flow]
What happened: Opening a workspace can present auth prompt even when user did not invoke a GitHub action, interrupting unrelated activity.
The requirement: Startup verification remains non-interactive unless explicitly user-triggered operation requires auth.
How to verify: Simulate startup with missing session/token and assert no UI prompt.

### Scenario 6: Contract Test Drift

Requirement tag: [Req: inferred — from skipped stale extension tests]
What happened: Activation tests are skipped and command IDs in assertions are stale, allowing registration regressions to pass CI.
The requirement: Activation contract tests execute and bind to current package contributions.
How to verify: CI fails when contributed command is missing from runtime registration.

### Scenario 7: Repeated Startup Churn

Requirement tag: [Req: inferred — from startup deployer flow]
What happened: Startup verification can rewrite files repeatedly even when content is unchanged, causing noisy VCS diffs and race pressure with manual edits.
The requirement: Startup writes are deterministic and skip no-op rewrites.
How to verify: Run startup verification twice on unchanged workspace and assert no write timestamps changed.

### Scenario 8: Over-scoped Authentication Requests

Requirement tag: [Req: inferred — from auth helper defaults]
What happened: Broad default scopes requested for read-only flows increase blast radius if token is compromised.
The requirement: Scope requests map to least privilege by operation.
How to verify: Assert startup/read flows request read scope while mutation flows request elevated scope only when required.

## AI Session Quality Discipline

1. Read QUALITY.md and REQUIREMENTS.md before proposing fixes.
2. Validate every bug claim with file:line evidence and an executable regression test plan.
3. Avoid replacing parse failures with silent reset behavior.
4. Treat command execution boundaries as security-sensitive surfaces.
5. Keep startup non-disruptive: no unsolicited interactive prompts unless operation demands it.
6. Do not delete existing scenarios; add and refine with evidence.

## The Human Gate

- Scope policy decisions for GitHub auth permissions.
- UX decisions on prompt timing and wording.
- Tradeoffs between strict startup checks and perceived responsiveness.
- Security review of shell command invocation semantics.
- Backward compatibility expectations for managed-file behavior.
