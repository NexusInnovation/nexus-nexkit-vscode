## Bootstrap (Read First)

1. quality/QUALITY.md
2. quality/REQUIREMENTS.md
3. quality/CONTRACTS.md
4. docs/architecture.md
5. quality/EXPLORATION.md

## Pass 1: Structural Review

Check for correctness bugs only (no style/refactor requests).

### Mandatory scrutiny areas

1. Input validation and boundary handling.
2. Resource lifecycle on all error/early-return paths.
3. Concurrency and state management consistency.
4. Unit/encoding correctness for parsed and persisted values.
5. Enumeration and whitelist completeness (mechanical two-list comparison where applicable).

### Guardrails

- Every finding needs file:line citation.
- Read function bodies, not signatures only.
- If uncertain, classify as QUESTION.
- Grep/search before claiming absence.

## Pass 2: Requirement Verification

Evaluate every requirement REQ-001..REQ-012 from quality/REQUIREMENTS.md.

For each requirement report:

- Status: SATISFIED / VIOLATED / PARTIALLY SATISFIED / NOT ASSESSABLE
- Evidence: one or more concrete file:line citations
- Analysis: expected behavior vs implemented behavior

## Pass 3: Cross-Requirement Consistency

Evaluate shared concepts across requirements:

- Activation lifecycle idempotency: REQ-001, REQ-002, REQ-012
- Managed file safety/encoding: REQ-004, REQ-005
- MCP merge safety: REQ-006, REQ-007
- Command execution integrity: REQ-008, REQ-009
- Quality contract enforcement: REQ-010, REQ-011

For each concept provide CONSISTENT or INCONSISTENT with evidence.

## Combined Summary

Produce a final table:

| Source | Finding | Severity | Status |
| ------ | ------- | -------- | ------ |

Then include totals by severity and a release recommendation: SHIP / FIX BEFORE MERGE / BLOCK.

## Closure Mandate

Every confirmed BUG must have:

- Regression test in quality/test_regression.\* with bug ID trace
- Or explicit exemption rationale with why executable regression test is not feasible

Use expected-failure idioms by language as needed; do not leave BUG findings without closure artifacts.
